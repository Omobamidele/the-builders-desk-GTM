# Contributing

This repo grows because people building GTM/AI systems share what they made. You don't need to be an expert, and you don't need permission, you need something real (even small) and five minutes to format it.

## What we're looking for

- Templates you actually use (decision logs, scorecards, docs, briefs)
- Cold email copy, sequences, or deliverability fixes that worked (or clearly didn't, and why)
- Prompts that solve a specific GTM/RevOps problem
- Automation recipes (Clay, n8n, Zapier, Make) with enough detail someone else could rebuild it
- Agent designs or example builds, even rough ones
- SQL queries or schema examples relevant to GTM/RevOps
- Corrections, improvements, or better versions of anything already in here

If you're not sure it fits, open an issue and ask. Worst case we point you to a better spot for it.

## Quality bar (low friction, not no bar)

Before you submit, check:

- [ ] It's usable by someone who isn't you... placeholders instead of your actual company data, clear enough to follow without extra context
- [ ] It's in Markdown and renders cleanly on GitHub (headers, tables, checkboxes — preview it before you submit)
- [ ] It follows the naming convention: lowercase, kebab-case, in the right category folder
- [ ] If it's a template, it goes in that category's `templates/` folder; if it's a real filled-in example, it goes in `examples/`
- [ ] No confidential info, no client names, no live API keys or credentials, scrub before you push

That's it. It doesn't need to be polished but It needs to be real and usable.

## How to submit

1. Fork the repo
2. Add your file in the right category folder, following the naming convention
3. If you're adding a new category or subfolder, add a one-line README explaining what it's for
4. Open a pull request with a short description: what it is, and what problem it solves
5. Tag it with the category name (e.g. `cold-email`, `prompting`, `sql`)

Don't have a GitHub workflow down yet? Open an issue instead, paste your content in, and someone will help you get it in properly. First-time contributors are welcome, we'd rather help you PR your first thing than have you not contribute at all.

## Review process

PRs get reviewed for the quality bar above, not for being "the best version." Small, honest, and useful beats polished and generic. If something needs changes, you'll get specific feedback, not a silent close.

## Recognition

Contributors get added to [`community/contributors.md`](./community/contributors.md). If your submission gets featured in the weekly update, you'll get a shoutout there too.
